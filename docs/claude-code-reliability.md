# Claude Code 可靠性工作流

这份文档不是让 Claude “更聪明”，而是让它**更不容易假通过、少漂移、少自说自话**。

适用场景：
- 长对话后质量下滑
- 看起来“已经完成”，但真实验证不足
- 急着交付绿色结果，跳过根因分析和真实路径测试
- `CLAUDE.md` 写了很多规则，但执行并不稳定

## 核心原则

1. **一次会话只做一个子任务**
   探索、规划、实现、验证，尽量不要混在同一长会话里。
2. **没有证据，不算完成**
   “已修复”“应该可以”“我验证过了”都不算，除非附带命令、测试结果、截图或可复现路径。
3. **测试的目的是发现问题**
   优先真实用户路径、真实输入、真实 URL、真实失败模式；避免只跑“最容易变绿”的检查。
4. **同类偏航纠正两次以上，就重开**
   一旦同类问题已经纠正两次，继续在原上下文里拉扯，通常比新开会话更差。
5. **实现者与验证者分离**
   能分开就分开：一个会话/agent 实现，另一个 fresh context 会话/agent review 或验收。

## 推荐工作流

### 1. 定义任务

先把任务压缩成 4 行：

```md
目标：
范围外：
验收标准：
验证方式：
```

示例：

```md
目标：修复 feed 页 URL 导入失败的问题。
范围外：不改 UI 文案，不做重构。
验收标准：粘贴 3 个真实 URL，均能成功入库并显示摘要。
验证方式：运行目标测试 + 手动走一次真实页面导入流程。
```

### 2. 先探索，再规划，再实现

推荐拆成三段：
- `探索`：只看代码、日志、数据流，先定位根因
- `规划`：列出要改哪些文件、怎么验证、有哪些风险
- `实现`：只做已批准的小步修改

如果任务复杂，先让 Claude 写简短计划，再让它按计划一步一步做。

### 3. 小步执行

每一小步都要求 Claude 按这个格式回报：

```md
这一步要改什么：
为什么要改：
怎么验证：
风险是什么：
```

不要让它直接“大包大揽改完再说”。

### 4. 强制验证

每次实现后，要求它给出：

- 执行了哪些命令
- 哪些测试通过/失败
- 是否做了真实路径验证
- 还有哪些没验证到

推荐固定句式：

```md
请不要先宣布完成。
先给我：
1. 你改了什么
2. 你怎么验证的
3. 哪些地方还没验证
4. 如果现在合并，最大风险是什么
```

### 5. 独立复核

对重要改动，再开一个 fresh context：

- 只给它任务目标、验收标准、改动文件
- 让它站在 reviewer 角度找漏洞
- 不把原会话的大量思路一起喂进去

推荐句式：

```md
不要实现功能。
只做 review：
1. 这次改动是否真的满足验收标准
2. 是否有假验证或跳过真实路径
3. 是否存在未覆盖的边界情况
4. 如果要补一项验证，最该补什么
```

### 6. 会话收尾

结束前强制输出：

```md
已修改文件：
已执行验证：
未验证项：
当前风险：
下一步建议：
```

这样即使重开会话，也不会只剩一句“应该搞定了”。

## 什么时候必须重开会话

出现以下任一情况，优先 ` /clear ` 或新开：

- 同一类错误已经纠正两次
- Claude 开始重复之前已经被否定的方案
- 开始“看起来很忙”，但没有新增证据
- 明显回避真实测试，只跑容易通过的检查
- 会话已经很长，而当前任务仍在扩 scope

## 推荐提示词模板

### 模板 A：开始任务

```md
这次只做一个子任务，不要顺手扩 scope。

目标：
[填写]

范围外：
[填写]

验收标准：
[填写]

验证方式：
[填写]

先不要写代码。
先做探索和计划，告诉我：
1. 根因假设
2. 要改哪些文件
3. 怎么验证
4. 最大风险
```

### 模板 B：开始实现

```md
按计划只执行当前这一步。
改动前先告诉我：
1. 这一步具体改什么
2. 为什么是最小改动
3. 你打算怎么验证

完成后不要先说 done。
先给出验证证据和未验证项。
```

### 模板 C：强制真实验证

```md
不要只跑最容易通过的检查。
优先真实用户路径、真实输入、真实 URL、真实失败场景。
如果你无法完成真实验证，必须明确写出：
- 没验证到什么
- 为什么没验证
- 风险是什么
```

### 模板 D：要求停止漂移

```md
你已经在同类问题上偏航两次了。
现在停止继续扩展方案。
请只做三件事：
1. 总结当前真实状态
2. 列出已验证与未验证项
3. 给出最小下一步
如果需要，建议我新开一个会话。
```

## 这套项目里的机械防护

本仓库已经增加以下 guardrails：

- `.claude/hooks/destructive-guard.sh`
  阻断 `rm -rf`、`git reset --hard`、`git clean -fd`、`git push --force`
- `.claude/hooks/rule-reminder.sh`
  每次 session 启动/恢复/compact 后，重新注入核心执行纪律
- `.claude/hooks/session-health.sh`
  统计工具调用次数，长会话时提醒只做收尾，不要继续扩 scope
- `.claude/hooks/verification-reminder.sh`
  记录本轮会话的编辑文件；如果后续还没看到新的验证命令，会在下一次提问前提醒先补验证或明确披露未验证项
- `.claude/settings.json`
  注册上述 hooks

## 最后的判断标准

真正好的 Claude Code 会话，不是“回复很快、看起来很忙、最后说 done”，而是：

- 边界清楚
- 证据充分
- 愿意承认未验证项
- 出现偏航时尽快停手
- 能在新会话里无损接力
